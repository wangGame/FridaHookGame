# HookLearn 案例执行模板清单

更新时间：2026-04-29

## 使用说明

本文把仓库中的核心案例统一成四段模板：

- 前置条件
- 执行步骤
- 预期输出
- 常见问题

建议按编号顺序练习，先保证链路跑通，再做定制化 Hook。

## 案例 01：Frida server 启动与连通

对应文档：`doc/1.Frida-server使用.md`

前置条件：

- 已连接 Android 设备，`adb devices` 可见
- 设备端可执行 `frida-server`

执行步骤：

1. 查看设备架构：`adb shell getprop ro.product.cpu.abi`
2. 推送 server：`adb push frida-server /data/local/tmp/`
3. 授权执行：`adb shell chmod 777 /data/local/tmp/frida-server`
4. 在设备端启动：`adb shell /data/local/tmp/frida-server`
5. 主机侧验证：`frida -U -n system_server`

预期输出：

- 能成功附加到 `system_server` 或至少看到附加行为

常见问题：

- 报权限问题：确认设备 root 权限与执行路径
- 架构不匹配：按 ABI 更换对应版本 `frida-server`

## 案例 02：枚举设备进程

对应文档：`doc/2.查看所有的进程.md`
对应脚本：`src/py/ListProcess.py`

前置条件：

- Frida server 正常运行
- Python 环境安装 `frida`

执行步骤：

1. 运行 Python 脚本枚举进程
2. 记录目标游戏包名或进程名

预期输出：

- 控制台打印 `PID + Name` 列表

常见问题：

- 枚举失败：检查 USB 连接和 `frida.get_usb_device(timeout=...)`

## 案例 03：Spawn 冷启动注入

对应文档：`doc/4.Spawn模式的使用.md`

前置条件：

- 已明确目标包名（如 `com.oakever.jigsawcard`）
- JS 脚本路径有效（如 `src/hook/UnityHookJs.js`）

执行步骤：

1. `device.spawn([package_name])`
2. `device.attach(pid)`
3. `session.create_script(js_code)`
4. `script.load()`
5. `device.resume(pid)`

预期输出：

- 出现 `Script injected`、`App resumed`、`Hook OK`

常见问题：

- 无日志输出：确认 `script.on("message", on_message)` 已绑定
- 启动即闪退：先注释激进 Hook，再逐步恢复

## 案例 04：模块与导出函数 Hook

对应文档：`doc/5.HookJs.md`
对应脚本目录：`src/js`、`src/hook`

前置条件：

- 目标 so 已加载（可先 `Process.findModuleByName`）

执行步骤：

1. 用 `Module.findExportByName` 定位导出符号
2. `Interceptor.attach` 拦截 onEnter/onLeave
3. 打印参数、返回值、调用栈

预期输出：

- 能看到指定函数被调用和参数信息

常见问题：

- 导出地址为 null：符号不存在或尚未加载
- 频繁崩溃：先减少打印量，避免在高频函数做复杂逻辑

## 案例 05：Il2Cpp 基础信息采集

对应文档：`doc/3.可以得到那些东西.md`
对应脚本：`src/dump_il2cpp.ts`、`src/dump_il2cpp.js`

前置条件：

- 游戏为 Unity/Il2Cpp 架构
- `libil2cpp.so` 可见

执行步骤：

1. 获取模块信息：`Process.getModuleByName("libil2cpp.so")`
2. 枚举导出符号：`Module.enumerateExportsSync`
3. 关注 `il2cpp_class_*`、`il2cpp_method_*` 系列 API

预期输出：

- 得到 il2cpp 基址、大小、导出函数统计

常见问题：

- 目标符号缺失：版本裁剪或符号被处理，可换 bridge 路线

## 案例 06：frida-il2cpp-bridge 编译与加载

对应文档：`doc/7.Frida-brige.md`、`doc/附1环境配置.md`
对应脚本目录：`src/brige`

前置条件：

- 已安装依赖：`frida-il2cpp-bridge`、`@types/frida-gum`、`@types/node`

执行步骤：

1. 编写 TS 脚本（如枚举类、枚举方法）
2. 编译：`frida-compile .\src\dump_il2cpp.ts -o .\src\dump_il2cpp.js`
3. Python 侧 attach 后先加载 bridge，再加载业务脚本

预期输出：

- 成功打印 Unity 版本、程序集、类、方法信息

常见问题：

- `Il2Cpp` 未定义：桥未正确加载或加载顺序错误

## 案例 07：指定类指定方法拦截

对应文档：`doc/9.指定类的指定方法.md`
对应脚本参考：`src/brige/HookClassMethod.ts`

前置条件：

- 已确认程序集名、类名、方法名

执行步骤：

1. 定位类：`Il2Cpp.domain.assembly(...).image.class(...)`
2. 定位方法：`klass.method("Create")`
3. 取 `method.virtualAddress` 后做 `Interceptor.attach`

预期输出：

- 方法被调用时稳定打印 this 或参数信息

常见问题：

- 地址为空：方法可能为泛型/抽象/未实例化，需换入口点

## 案例 08：方法调用链追踪

对应文档：`doc/10.追踪方法调用.md`
对应脚本参考：`src/brige/Trace.ts`、`src/index.ts`

前置条件：

- 目标类可定位（如 `Assembly-CSharp.Game`）

执行步骤：

1. 通过 bridge 拿到目标类对象
2. 使用 `Il2Cpp.trace(true).classes(Target).and().attach()`
3. 在游戏内触发功能，观察调用顺序

预期输出：

- 输出方法进入、参数、返回等追踪日志

常见问题：

- 日志过多：先限定类或方法白名单再追踪

## 案例 10：导出 JSON + 生成 IDA 重命名脚本

对应文档：`doc/附导出json.md`
对应脚本目录：`src/dumpIlp2dumperSo`

前置条件：

- 能稳定拿到方法 VA 与 RVA
- 本地有写权限输出目录

执行步骤：

1. JS 侧批量收集 namespace/class/method/rva
2. Python 侧 `on_message` 接收 batch 并累积
3. 落地 `il2cpp_dump.json`
4. 同时生成 `ida_rename.py`

预期输出：

- 输出总量增长日志
- 最终得到 JSON 与 IDA 脚本文件

常见问题：

- 重名冲突：提前做名字清洗和去重
- 空地址较多：过滤无效 `virtualAddress`
